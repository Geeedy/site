# Web Application Development: A Product, Not a Presentation

A web application differs from a website in that logic lives in it: cabinets, roles, calculations, integrations. We build such products by engineering rules: an MVP in a reasonable time, iterations by data, architecture for the task, not for fashion. We use microservices where they are justified, not because people write about them.

## Monolith or microservices

[INFOGRAPHIC: ../assets/infographics/veb-prilozheniya-2.svg | Monolith or microservices]

| | Monolith | Microservices |
|---|---|---|
| Start | Faster and cheaper | More expensive upfront |
| Team | One | Independent teams |
| Scaling | As a whole | Per service |
| When to choose | MVP, one product | Ecosystem, high load, AI modules |

Our position is honest: most products at the start need a careful monolith with clean module boundaries, because it is cheaper and faster. Microservices are justified when there are independent loads, teams, or AI modules, for example in a [microservice AI ecosystem](/uslugi/razrabotka-saitov/) where the site, agents, and analytics live as separate services. We design the monolith so that splitting into services later does not require a rewrite.

## How we use AI in development

Per industry data, [about 90% of developers use AI agents](https://convergent.digital/blog/web-development-trends-2026) to generate a code skeleton with later refinement. We work the same way: the machine speeds up the typical; engineers design architecture, write critical logic, and own quality. For you that is speed without losing reliability, because code goes through review and tests regardless of who typed the first version.

## AI inside your product

A web application is a natural place for AI features: an assistant in the cabinet, auto document processing, smart search over data, [agents](/uslugi/ii-agenty/) for users’ routine operations. We leave room for these capabilities in the architecture from day one, so adding an AI module later is a matter of weeks, not a rewrite.

## Security and load

Roles and permissions from day one, action audit, encryption of sensitive data, personal data processing under 152-FZ. We run load testing before production, because the product must survive both marketing success and a TV spot.

## FAQ

**How much does a web application cost?**
Priced individually by scenarios, roles, and integrations; each project has its own configuration. We give an MVP estimate after short design. Design documentation stays with you either way; that is standalone value.

**MVP or a full version at once?**
MVP, that is the core of value in minimal time, then iterations by real usage data. A “full version at once” almost always means a year of development and features nobody needs. Data is cheaper than assumptions.

**Monolith or microservices?**
For a start, more often a monolith with clean boundaries; it is cheaper and faster. Microservices are justified with independent loads, teams, or AI modules. We design so that growth does not require a rewrite.

**Which stack do you use?**
The stack is chosen for the task and your team. Criteria: technology maturity, availability of specialists on the market, maintainability. We do not use exotic tech for its own sake.

**Who owns the code?**
You. Repository, documentation, and access are handed over fully. Any team can develop the product further, because freedom from vendor lock-in is part of our engineering culture.

**How do you embed AI in the product?**
Cabinet assistants, auto document processing, smart search, agents for user routine. We leave room in the architecture from the start. More on capabilities on the [AI agents](/uslugi/ii-agenty/) page.

**How long does an MVP take?**
By market from 3 months depending on scenario volume. Exact timeline is locked after design. Demos every 1–2 weeks; you see progress instead of waiting for a final surprise.

**How do support and development work?**
Monitoring, incidents by SLA, development iterations from the backlog, format by agreement. The product is handed over so your team can support it too: documentation and clean code are standard.

**Will the app handle load growth?**
Load testing is part of launch; architecture is designed for stated peaks with headroom. If you expect explosive growth, we plan horizontal scaling, but we say honestly when that is premature.

**You use AI in development: is that safe?**
Yes. AI generates skeletons and typical code; engineers design, review, and test. Sensitive code and client data do not go through external AI services. Quality is guaranteed by process, not faith in the machine.
